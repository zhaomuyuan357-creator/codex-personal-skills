# Session Closeout

## 目的

这个 skill 用在一次工作会话结束时，帮助把项目收尾到“干净、可续接、不过度更新”的状态。

## 核心原则

- 不要默认代码一定要提交，更不要默认一定要推送。
- 优先更新短期项目记忆，再判断是否需要改长期说明。
- 交接内容要高信号、低废话，避免写成长篇流水账。
- 尊重当前项目已有的进度文档、handoff 文档、任务文档和仓库习惯。

## 收尾流程

1. 检查当前工作区状态。  
   查看 git 状态、变更文件，以及项目当前已有的 memory / handoff / progress 文档。

2. 判断代码是否适合提交。  
   只有在改动已经形成明确检查点、用户明确要求，或当前会话确实适合落一个仓库快照时，才提交。

3. 判断是否适合推送。  
   只有在远端工作流存在、当前分支适合同步、并且没有明显理由需要先保留本地时，才推送。

4. 总结本次会话结果。  
   至少提炼出：
   - 已完成内容
   - 进行中内容
   - 当前阻塞或风险
   - 下一步最高优先级

5. 更新项目记忆。  
   优先更新项目的短期运行记忆，例如：
   - 进度日志
   - handoff 文档
   - 任务追踪文档
   - 检查点说明

6. 仅在必要时更新长期说明。  
   只有当以下内容真的发生变化时，才更新 `AGENTS.md`、贡献说明、工作流说明等长期文档：
   - 新对话续接方式
   - 当前阶段判断
   - 稳定工作规则
   - 重要仓库注意事项

7. 生成下一次新对话的续接提示词。  
   这段提示词应尽量让新对话快速进入状态，至少包括：
   - 先读什么
   - 当前基线是什么
   - 当前正在做什么
   - 有哪些重要注意点
   - 下一步先做什么

8. 向用户汇报收尾结果。  
   要明确说明：
   - 是否已 commit
   - 是否已 push
   - 更新了哪些记忆文件
   - 是否更新了长期说明文件
   - 可直接复制的新对话续接提示词

## 判断规则

### 什么时候适合 commit

适合提交的情况：

- 这次改动已经形成一个明确检查点
- 改动可以被理解为一个完整单元
- 用户明确要求做仓库收尾，或当前确实是结束会话的自然节点

不适合提交的情况：

- 当前改动仍然半坏、半试验、没有形成稳定节点
- 无关改动混在一起，还没有梳理清楚

### 什么时候适合 push

适合推送的情况：

- 用户明确要求 push
- 当前分支本来就应该同步到远端
- 当前仓库状态适合共享，不需要先本地保留

不适合推送的情况：

- 没有配置远端
- 当前分支明显只是本地实验分支
- 用户只想要本地检查点，不想立刻同步

### 什么时候更新项目记忆

优先写进去的应该是：

- 当前真实进展
- 当前阻塞点
- 明确下一步

避免：

- 把整个项目历史重写一遍
- 在多个文档里无意义重复同样的话

## 输出模板

最终输出通常应包含这几个部分：

```text
收尾结果：
- Commit: [已完成 / 未完成]
- Push: [已完成 / 未完成]
- 更新的项目记忆： [文件列表]
- 更新的长期说明： [文件列表或无]

当前状态：
- 已完成：
- 进行中：
- 阻塞或风险：
- 下一步优先级：

新对话续接提示词：
[可直接复制的提示词]
```

## 触发示例

- “做一次收尾。”
- “帮我做个收尾。”
- “更新交接内容，再给我一个新对话续接提示词。”
- “检查一下现在该不该 commit / push，然后同步项目记忆。”
